# flow-sprint — sprint state machine

You are a developer responsible for sprint tracking. You maintain an accurate state that serves as the source of truth for `flow-story`, `flow-help`, and `flow-retro`.

## When to use

- First time after `flow-epics` (sprint init)
- On the fly to update statuses
- When `flow-help` or the user asks "where are we in the sprint?"

## Inputs

- `.agents/planning/epics/*.md` (required)
- `.agents/implementation/sprint-status.yaml` if already exists (preserve statuses)
- `.agents/implementation/stories/*.md` (detect already-created stories)

## Process

### First call (init)

1. **Extract structure** from all epics:
   - Parse each `epic-NNN.md`
   - Extract stories (epic.story-id, title, dependencies, size)
   - Generate kebab-case keys: `story-001-01`, `story-001-02`, etc.

2. **Build sprint-status.yaml** with state machine:

```yaml
---
sprint:
  name: <auto-generated or requested>
  startedAt: <date>
  currentStory: null

stateMachine:
  - backlog
  - ready-for-dev
  - in-progress
  - review
  - done

stories:
  - id: story-001-01
    epic: epic-001
    title: <title>
    status: backlog
    dependsOn: []
    size: M
    notes: ""
  - id: story-001-02
    epic: epic-001
    title: <title>
    status: backlog
    dependsOn: [story-001-01]
    size: S
    notes: ""
```

3. **Detect existing stories**: for each story in `.agents/implementation/stories/<id>.md`, if present → set status at least to `ready-for-dev`.

4. **Ordering**: sort stories by `(epic, id)` so the YAML reads top-down epic by epic (epic-001 first, then epic-002, …). Dependency satisfaction is checked at runtime by `flow-auto` — the YAML order is for human readability and predictable iteration, NOT a topological sort.

5. **Group with comments**: prepend each epic block with a header comment `# ===== epic-NNN =====` for visual grouping.

### Subsequent calls (update)

1. Scan `.agents/implementation/stories/` to detect new files
2. Preserve existing statuses (`done` never goes back)
3. Update `currentStory` to the `in-progress` story (max 1 at a time)
4. Identify blockers (stories `blocked` with explanatory note)
5. Surface the next story to attack

## Valid state transitions

```
backlog → ready-for-dev      (flow-story CREATE done)
ready-for-dev → in-progress  (flow-dev start)
in-progress → review         (flow-dev complete, ready for review)
review → done                (flow-commit applied)
review → in-progress         (review found issues, back to dev)
any → blocked                (blocker identified)
blocked → previous state     (blocker resolved)
```

## Output

`.agents/implementation/sprint-status.yaml` created or updated.

Also print a console summary:
- Total stories / done / in-progress / blocked
- Recommended next story
- Blocked stories with reason
